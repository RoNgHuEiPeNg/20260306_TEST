## 1. 簡介

### 1.1 文件目的
本文件旨在詳細定義「跨部門專案管理與成本控制系統」之最小可行產品 (MVP) 的功能範疇、業務邏輯、用戶體驗及技術要求。此文件將作為軟體開發團隊（Frontend, Backend, QA）與利益相關者（HR, 工程主管）之間的通訊基準，確保產品開發符合業務價值。

### 1.2 產品背景
目前公司內部的人力資源團隊（擔任 PM 角色）與軟體開發團隊在協作上存在資訊斷層。HR 需要掌握專案的財務成本與人力投資報酬率 (ROI)，而開發團隊則專注於任務執行。現有的工具（如試算表或單純的 To-do list）無法自動化地將「開發工時」轉化為「財務成本」，且難以處理敏感的薪資費率權限問題。因此，需要一個整合性的系統來填補這一空白。

### 1.3 產品願景
打造一個「成本驅動」的輕量化協作平台，實現專案進度與財務數據的即時同步。透過自動化的工時成本計算引擎，讓管理層能基於數據進行精準的資源調度與決策，同時降低開發人員行政作業的負擔。

### 1.4 MVP 目標
*   **建立集中化的專案庫**：取代散亂的試算表，記錄專案背景與預算。
*   **實現自動化成本計算**：工時回報後立即根據成員費率計算專案支出。
*   **確保財務數據安全性**：透過嚴格的權限拆分，隱藏開發人員不應看到的敏感費率。
*   **保留費率變更軌跡**：確保費率調整後，歷史成本紀錄不被錯誤回溯。
*   **視覺化資源負載**：提供儀表板讓 PM 快速識別人力分配是否過度或不足。
*   **產出標準化財務報表**：支持一鍵匯出 Excel，滿足月度結算需求。

## 2. 目標用戶

### 主要用戶
1.  **HR / 專案經理 (PM)**：負責開案、預算控管、費率設定與成本分析。
2.  **軟體開發人員 (Developer)**：負責執行任務、回報進度與實際投入工時。

### 用戶特徵
*   **HR / PM**：
    *   重視數據準確性與報表輸出。
    *   具備基本的專案管理知識，但可能不熟悉複雜的敏捷開發術語。
    *   對成本超支高度敏感。
*   **開發人員**：
    *   追求簡單、直覺的輸入介面，厭惡複雜的行政作業。
    *   需要清楚知道自己的任務優先順序。
    *   技術能力高，但對公司財務邏輯不一定有深入了解。

## 3. 核心問題與解決方案

*   **核心問題 1：專案成本統計滯後且易出錯**
    *   **MVP 解決方案**：建立自動化計算邏輯。當開發者填寫工時後，系統即刻抓取後台維護的「時薪費率」進行乘積運算，並自動更新專案總成本，確保數據實時性。
*   **核心問題 2：薪資費率敏感資訊外洩風險**
    *   **MVP 解決方案**：實施角色權限隔離 (RBAC)。後端 API 根據角色過濾敏感欄位，確保開發人員端僅能看到「工時數」，而「費率」與「總金額」僅對 HR/PM 角色開放。
*   **核心問題 3：費率調整導致歷史成本數據混亂**
    *   **MVP 解決方案**：採用「成本快照 (Snapshot)」機制。每筆工時紀錄在提交時，會將當時的費率與計算結果直接寫入該紀錄中，而非動態關聯人員表，確保歷史費率的完整性。
*   **核心問題 4：資源分配不均，難以識別閒置人力或過載人員**
    *   **MVP 解決方案**：提供「資源維度儀表板」。統計每位成員在特定時間範圍內的總投入工時，並以視覺化圖表呈現資源負載概況。

## 4. 產品功能 (MVP 範圍)

### 4.1 專案管理模組 (Project Management)
*   **功能描述**：提供 HR/PM 建立與維護專案核心資訊的中心。
*   **具體子功能**：
    *   **專案建立表單**：包含專案名稱、專案說明、效益說明（商業價值）、起訖日期。
    *   **預算與優先級**：設定專案預算上限及優先順序（高、中、低）。
    *   **專案列表導覽**：支援按優先順序、進度或剩餘預算進行排序與篩選。
    *   **進度追蹤器**：根據已完成任務與總任務數自動計算百分比。

### 4.2 任務清單模組 (Task Management)
*   **功能描述**：輕量化的任務追蹤系統，供團隊日常執行使用。
*   **具體子功能**：
    *   **任務建立**：包含名稱、描述、負責人、優先順序。
    *   **預計工時設定**：由 PM 或開發者填寫預計花費的小時數。
    *   **狀態流轉**：提供「待處理 (To Do)」、「執行中 (In Progress)」、「已完成 (Done)」三種狀態切換。
    *   **任務關聯**：所有任務必須掛載於特定專案下。

### 4.3 工時追蹤模組 (Time Tracking)
*   **功能描述**：開發人員回報實際工作量的入口。
*   **具體子功能**：
    *   **實際工時錄入**：開發者在特定任務下輸入所花費的小時數（支援至小數點後一位）。
    *   **自動生效機制**：提交後立即觸發成本運算逻辑，無需手動審核。
    *   **個人工時紀錄表**：用戶可查看自己過去一週或一個月的所有工時紀錄，並在未結算前進行修改。

### 4.4 HR 後台費率管理 (Rate Management)
*   **功能描述**：管理人員薪資費率，為成本計算提供基礎數據。
*   **具體子功能**：
    *   **人員名冊維護**：列出系統內所有帳號及其對應角色。
    *   **費率設定**：以「人」為單位設定每小時時薪 (Hourly Rate)。
    *   **變更生效紀錄**：記錄費率修改時間點，系統依此判斷新舊費率的適用區間。
    *   **權限管控**：此模組僅對 HR 角色完全開放，其餘角色無法存取 API 節點。

### 4.5 財務與資源分析儀表板 (Analytics Dashboard)
*   **功能描述**：將數據轉化為決策資訊。
*   **具體子功能**：
    *   **預算執行率圖表**：以專案維度顯示「已花費成本 vs 預算上限」的燃盡圖或進度條。
    *   **工時偏差分析**：比較「預計工時」與「實際工時」的差異，識別專案風險。
    *   **資源負載概況**：顯示團隊成員的工時佔比，預防單一成員過載。
    *   **高優先級專案監控**：快速篩選優先級為「高」的專案當前狀態。

### 4.6 報表匯出模組 (Report Export)
*   **功能描述**：滿足外部稽核或月度會議需求。
*   **具體子功能**：
    *   **專案成本總覽匯出**：匯出 CSV/Excel，包含所有專案的預算、實際支出、進度。
    *   **成員工時明細匯出**：針對特定人員或專案，產出詳細的工時清單與日期。

### 4.7 權限與安全 (Security & Auth)
*   **功能描述**：確保數據安全性。
*   **具體子功能**：
    *   **RBAC 角色控管**：定義 HR/PM 與 Developer 兩套核心權限集。
    *   **敏感欄位遮罩**：開發者介面全局屏蔽任何金額相關欄位。
    *   **登入驗證**：基本的電子郵件與密碼登入機制。

## 5. 用戶故事 (User Stories)

1.  **作為 HR (PM)**，我想要在建立專案時填寫**效益說明**，以便我能向高層解釋投入這些資源的商業價值。
2.  **作為 HR (PM)**，我想要設定專案的**優先順序**，以便在資源有限時決定哪些任務該先行啟動。
3.  **作為 HR**，我想要在後台維護每個人的**時薪費率**，以便系統能自動計算專案的人力成本。
4.  **作為 HR**，當我調整成員費率時，我希望**舊的工時紀錄維持舊費率**，以便財務結算的歷史紀錄保持準確。
5.  **作為開發人員**，我想要在任務下直接輸入**實際工時**，以便我不需要手動計算每天花在專案上的時間。
6.  **作為開發人員**，我想要查看指派給我的**任務清單與優先級**，以便我能專注在當前最重要的工作中。
7.  **作為 HR (PM)**，我想要在儀表板看到**預算執行率**，以便在成本接近上限時及時發出警示。
8.  **作為 HR (PM)**，我想要一鍵匯出 **Excel 成本報表**，以便我進行月度財務報告。
9.  **作為系統管理員**，我想要確保開發人員**看不到費率數據**，以符合公司薪資保密政策。
10. **作為 HR (PM)**，我想要比較**預計工時與實際工時**，以便在未來估算專案時有更好的數據參考。

## 7. 技術要求

### 7.1 後端框架
*   **建議**：Node.js (搭配 Express.js 或 NestJS)
*   **理由**：Node.js 具備極佳的 I/O 處理能力，適合頻繁的 API 調用；NestJS 提供強大的架構約束，適合需要處理財務計算等嚴謹邏輯的系統。

### 7.2 資料庫
*   **建議**：PostgreSQL (關聯式資料庫)
*   **理由**：專案、任務、工時與費率之間存在強烈的關聯性，PostgreSQL 的 ACID 特性能確保財務數據的一致性與交易安全。

### 7.3 資料庫 ORM
*   **建議**：Prisma
*   **理由**：提供型別安全 (Type-safety)，能減少在處理複雜成本計算邏輯時的程式碼錯誤，且具備直觀的 Schema 定義方式。

### 7.4 前端技術
*   **建議**：React.js (搭配 Vite 與 Tailwind CSS)
*   **理由**：React 生態系豐富，元件化開發有利於未來儀表板功能的擴充；Tailwind CSS 能快速構建專業且簡潔的管理後台 UI。

### 7.5 其他整合
*   **Excel 處理**：使用 `exceljs` 或 `xlsx` 套件實現後端報表生成。
*   **狀態管理**：使用 `TanStack Query (React Query)` 處理非同步數據獲取與快取。

### 7.6 部署環境
*   **建議**：Docker 容器化後部署於 AWS (ECS/App Runner) 或 Vercel (前端) + Render (後端)。
*   **理由**：確保開發與生產環境一致，並支持輕鬆擴展。

### 7.7 版本控制
*   **建議**：Git (使用 GitHub 或 GitLab)
*   **理由**：分支管理 (Git Flow) 以確保代碼質量，並整合 CI/CD 流程。

## 8. 成功指標

*   **財務準確度**：系統計算的專案成本與手動覆核的誤差率為 0%。
*   **行政效率提升**：HR/PM 產出月度成本報表的時間從「數天」縮短至「數分鐘」。
*   **用戶採用率**：開發團隊 100% 在系統內記錄工時，不再使用外部臨時工具。
*   **風險預警能力**：PM 能夠在專案超支發生前，提前至少一週從儀表板識別出預算風險。
*   **數據完整性**：費率調整後，歷史報表的數據一致性達到 100%。

## 9. 假設與約束

### 9.1 假設
*   用戶擁有穩定的互聯網連接以存取 Web 應用。
*   HR 提供的初始人員費率數據是準確的。
*   開發人員能自律地每天或每週回報工時，不涉及複雜的工時追蹤硬體（如打卡機）。
*   專案成本僅計算人力小時成本，不包含設備、場地等非人工支出。
*   系統主要在桌面瀏覽器上使用，不強求移動端瀏覽器的極致優化。

### 9.2 約束
*   **預算限制**：MVP 必須在既定的開發週期內完成，不包含任何外部 API 收費服務。
*   **安全性約束**：所有財務數據傳輸必須經過加密 (HTTPS)，且敏感數據不得存在客戶端緩存。
*   **合規性約束**：系統設計必須符合公司對員工個人薪資數據處理的隱私條款。
